# 引き継ぎ資料作成スキル (Handover Docs)

退職・異動・移管のときに、リポジトリを調査して 新しい担当者がこのプロジェクトを引き継げる だけの情報を 1 つの Markdown にまとめる。

## このスキルのゴール

「次の人が、これ 1 枚を読めば動かせて・直せて・困ったときに辿れる」状態を作る。
コードを読めば分かることより、コードを読んでも分からないこと（背景・経緯・暗黙知・運用・連絡先）を優先して残す。

## 進め方

### 1. リポジトリを調査する（サブエージェントに委譲）

調査は `Explore` サブエージェントに任せる。これはトークン節約のため: 大量のファイル本文はサブエージェント側の文脈で読ませ、メインには項目ごとの要点メモだけを返させる。こうすれば調査の網羅性（＝品質）はそのままで、メイン文脈の消費を大きく抑えられる。

サブエージェントに「以下を調べ、項目ごとに簡潔な箇条書きで返す。ファイル全文は貼らない」と指示する:

- 概要: `README*`, `docs/`, `CONTRIBUTING*`, `ARCHITECTURE*`, Wiki へのリンク
- 技術スタック・依存: `package.json`, `requirements.txt`, `pyproject.toml`, `go.mod`, `Gemfile`, `pom.xml`, `Cargo.toml` など
- 構成: トップレベルのディレクトリ構造と、各ディレクトリの役割
- エントリポイント: `main.*`, `index.*`, `app.*`, `cmd/`, ルーティング定義
- ビルド/実行/テスト: `Makefile`, `package.json` scripts, `Taskfile`, `tox.ini`, `docker-compose*`, `Dockerfile`
- CI/CD: `.github/workflows/`, `.gitlab-ci.yml`, `.circleci/`, `Jenkinsfile`, デプロイ設定
- インフラ/環境: `terraform/`, `k8s/`, `helm/`, `serverless.yml`, `*.tf`, IaC, 環境変数の定義（`.env.example` 等）
- 外部サービス連携: SDK・API クライアントの利用箇所、Webhook、cron/スケジューラ
- 設定・シークレットの「在りか」: `.env.example`, secrets manager 参照、`config/` （値そのものは資料に書かない）
- 既知の課題: `Issues`、未マージの重要 PR、ハマりどころのメモ

調査時の注意:
- 全ファイルは読まない。ディレクトリツリーは浅く（2階層程度）把握し、上記のアンカーファイル（README / マニフェスト / CI / インフラ設定 / `.env.example` / エントリポイント）だけ開く。個々の実装ファイルは、既知の問題やトラブルシューティングで具体的に必要な箇所だけ部分的に見る。
- シークレットや認証情報の値を資料に転記しない。「どこに保管されているか（例: AWS Secrets Manager の `prod/db`）」だけを書く。
- 推測で断定しない。確認できなかった項目は空欄にせず「要確認」と明示する。

### 2. ドラフトを生成する（テンプレをコピーして埋める）

トークン節約のため、テンプレ全体を出力し直さない。雛形をコピーしてからプレースホルダだけを編集する:

1. このスキルの `TEMPLATE.md` を、対象リポジトリ直下（または `docs/`）に `PROJECT_KNOWLEDGE_BASE.md` としてコピー（`cp`）する。
2. 各 `<...>` / `🖊 要記入` を、調査メモを元に `Edit` で置換していく。定型部分は触らない。
3. 案件に当てはまらないセクション・小見出しは丸ごと削る。

- コードから機械的に分かること（スタック、コマンド、構成）は埋める。
- 人にしか分からないこと（連絡先、関係者、社内ツールの権限、業務背景、運用の暗黙知）は、ユーザーに質問するか、`🖊 要記入` のプレースホルダを残す。
- 各項目で「確認できた事実」と「推測・要確認」を区別する。
- フォーマットに迷ったときだけ `examples/PROJECT_KNOWLEDGE_BASE.sample.md` を参照する（毎回は読まない）。

### 3. 分からないことはラベルで残す（基本は質問しない）

コードから分からないことを対話で聞き出さない。これはトークン節約のため: 質問→回答→反映の往復が無くなる。代わりに `🖊 要記入` / `⚠ 要確認` ラベルを置き、最後（手順4）にまとめて一覧で報告する。資料を書く本人が後でまとめて埋めればよい。

主にラベルで残る項目（コードからは分からない）:
- 案件体制（ディレクター・エンジニアの氏名・連絡先）
- インフラやアカウントの管理主体（社内 or 得意先）・契約/支払い
- 運用上の暗黙知（「公開後はCDNをinvalidateする」「月初にバッチを手動確認」等）

例外: その場で答えないと資料の骨格が決まらない場合だけ、`AskUserQuestion` で最小限まとめて聞く。

### 4. 仕上げて報告する

`PROJECT_KNOWLEDGE_BASE.md`（手順2で作成済み）を見直し、要確認 / 要記入 が残っている箇所を一覧でユーザーに報告する。

書式のルール（プレースホルダを埋めるときに守る。テンプレ自体は準拠済み）:
- 強調の太字（`**`）は使わない。構造は見出し（`#`）と箇条書き（ul）で表現する。
- 表（テーブル）は使わない。複数項目は箇条書きにし、項目内の属性は入れ子の箇条書きで表す。
- 強調したい語は文章で書く。

## 必要な項目（資料の構成）

以下の 3 つの大セクションに分けて構成する。詳細な雛形は `TEMPLATE.md` を参照。

① 案件概要 — この案件が何で、誰が関わっているか
- サイト概要（何のサイト/システムか・目的）
- 機能・ページ等
- 案件体制（ディレクター / エンジニア）
- インフラ環境（社内 or 得意先）
- 関連資料（Wiki と案件ドライブの2つのみ。他のリンクは載せない）

② 開発環境・セットアップ — 手元で動かして変更を出すまで
- リポジトリ構成
- 技術スタック
- フロントエンド設定
- バックエンド設定
- インフラ設定
- 開発セットアップ手順
- 環境変数
- ブランチ運用

③ 運用・保守 — 本番を動かし続けるために
- 環境URL一覧
- デプロイ手順
- 定常作業（必要に応じて）
- 既知の問題や注意事項等
- トラブルシューティング

> 案件に合わせて小見出しは増減してよい（例: フロント/バックが分かれていない案件なら統合、CMS が無ければ省略）。空のままより、不要なら見出しごと削るほうが読みやすい。
